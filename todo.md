# 单元测试与集成测试

# 单元测试概述

单元测试是软件开发中最小粒度的测试方式，主要针对软件代码中最小独立执行单元，例如业务层 Service 方法、工具类方法、核心业务逻辑片段进行独立化、隔离化的白盒测试。在若依前后端分离项目中，单元测试主要针对用户管理、角色管理业务层代码开展测试。

单元测试核心核心设计原则为隔离性与独立性，在测试过程中，通过 Mockito 模拟伪造数据访问层 Mapper、数据库操作等外部依赖，不连接真实 MySQL 数据库、不使用 Redis、不加载完整 Spring 容器，完全脱离外部硬件与中间件环境，只专注校验代码本身的业务逻辑。

本次单元测试基于 JUnit5 + Mockito 技术实现，单独对用户模块、角色模块的新增、修改、删除、查询核心业务逻辑进行逐一验证。通过手动构造模拟数据、伪造数据层返回结果，检测代码的判断逻辑、参数处理、返回值结果、基础规则校验是否符合业务设计要求。

单元测试的核心作用：

- 专注代码内部逻辑，提前发现空指针、逻辑错误、判断漏洞等代码级 Bug；
- 脱离数据库和第三方组件依赖，运行速度快、测试成本低；
- 单个模块独立测试，用户管理与角色管理互不干扰，问题定位精准；
- 保障代码修改后原有逻辑不受影响，便于后期迭代维护与回归测试。

单元测试只关注单一模块内部代码逻辑，不考虑多层级调用、模块之间协作、数据库持久化等外部交互问题，是保障代码质量的基础测试环节。

总之，单元测试就是用代码片段验证其他的代码片段，所以是需要写代码，主要由程序员自己完成。如果要生成测试用例文档，一般都是自动化生成，测试对象是最小的，不可再分的代码判断（如函数、类、方法等），但是函数出现嵌套，会用 mock，不直接链接外部。

# 集成测试概述

集成测试是在单元测试基础上的进阶测试，是将拆分独立的代码单元、多层架构组件、外部资源按照项目结构组合后，进行统一整体的联合测试。在若依前后端分离架构中，系统采用分层设计（Controller 控制器、Service 业务层、Mapper 数据层、MySQL 数据库），集成测试主要验证各层级之间、程序与数据库之间的交互有效性。

集成测试不再隔离外部环境，通过 `@SpringBootTest` 加载项目完整的 SpringBoot 容器，读取项目全部配置文件，注入真实的业务 Bean，连接真实 MySQL 数据库，完全模拟项目真实运行环境。

本次集成测试以完整功能模块为单位，分别对用户管理、角色管理进行全链路测试，完整模拟请求调用链路：控制器 → 业务层 → 数据层 → 数据库。通过执行真实的新增、查询、修改、删除操作，实现数据在数据库中的真实写入、读取、更新与删除，重点测试各层之间参数传递、方法调用、依赖注入、数据持久化、事务控制等协作能力。

集成测试的核心作用：

- 验证项目分层架构的合理性，确保各组件之间调用正常、参数传递无误；
- 校验程序与 MySQL 数据库的交互能力，确保数据可正常持久化存储与读取；
- 发现单元测试无法覆盖的问题，例如层级调用异常、配置缺失、数据库字段约束、资源加载失败等问题；
- 贴近实际生产运行环境，验证单个功能模块整体运行的稳定性与完整性。

集成测试聚焦单模块全链路整合，为后续多模块联动的系统测试打下基础，是介于单元测试和系统测试之间的关键测试环节。

主要分为两种：

1. 完整功能模块全链路调用（多个单元测试组成）
2. 功能模块代码（即部分链路）之间进行调用（一个调用另一个功能接口）

总之，集成测试就是用代码片段验证其他的代码片段，但是不同的是，测试对象是集成的（由各个单元组装在一起），所以是需要写代码，主要由程序员和测试人员完成。如果要生成测试用例文档，一般都是自动化生成。

# 四、单元测试

**目标**：只测试 Service 业务逻辑，不连接数据库、不启动完整容器

**技术**：JUnit5 + Mockito（模拟 Mapper 层，如果出现了调用其他层，也直接 mock 即可）

**逻辑**：伪造数据库层返回值，验证业务方法的参数、流程、返回结果是否正确

**特点**：速度极快、环境无关、专注代码逻辑

**逻辑说明**：

- 使用 `@Mock` 模拟数据库 Mapper 层，不访问真实数据库
- 使用 `@InjectMocks` 注入 Service，测试业务逻辑
- 手动指定模拟返回值，验证业务方法执行是否正常
- 纯代码逻辑校验，无环境依赖

**说明与注意**：

截图里的这三个注解，是 JUnit5 + Mockito 单元测试的核心工具，专门用来做「隔离式单元测试」。

1. `@ExtendWith(MockitoExtension.class)`
   - 作用：告诉 JUnit5，这个测试类要启用 Mockito 的功能，帮你自动初始化 `@Mock`、`@InjectMocks` 这些注解。
   - 为什么用？ 没有它，Mockito 的模拟对象和依赖注入就不会生效，测试会报错。

2. `@Mock`
   - 作用：创建一个「模拟对象」，代替真实的外部依赖。比如代码里的 `@Mock private SysUserMapper userMapper;`：
     - 这个 userMapper 不是真实的数据库 Mapper，而是 Mockito 伪造的假对象。
     - 你可以手动定义它的行为，比如 `when(userMapper.insert(user)).thenReturn(1);`，意思是 "当调用 insert 方法时，直接返回 1，不用真的去数据库执行 SQL"。
   - 核心目的：隔离外部依赖（比如数据库），让单元测试只专注 Service 层的业务逻辑，不受数据库、网络的影响。

3. `@InjectMocks`
   - 作用：创建被测试类的实例，并且把 `@Mock` 标记的模拟对象，自动注入到这个实例里。比如代码里的 `@InjectMocks private SysUserServiceImpl userService;`：
     - Mockito 会创建一个 SysUserServiceImpl 的实例。
     - 然后把上面 `@Mock` 创建的 userMapper，自动注入到 userService 里。
     - 这样 userService 在调用 userMapper 时，就会调用模拟的假对象，而不是真实的数据库。

**关于 `@InjectMocks` 的实现类能否换成接口**：

直接说结论：不能。

为什么不能？

- 接口不能被实例化：`SysUserService` 是一个接口，它只定义了方法签名，没有具体的实现逻辑。Mockito 的 `@InjectMocks` 需要创建一个对象实例，而接口本身无法被实例化，所以必须用它的实现类 `SysUserServiceImpl`。
- 单元测试的目标是「实现类的业务逻辑」都写在 `SysUserServiceImpl` 里，而不是接口里。单元测试的目的就是验证这些实现逻辑是否正确，所以必须把被测试的对象指定为实现类。

- 注入真实 Service，调用真实方法
